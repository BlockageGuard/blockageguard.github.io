Summary

Two related misconfiguration findings were identified on Ghost CMS 6.25 instances at blog.premiumalts.to and dev-blog.premiumalts.to.

F-156: The Ghost configuration flag allow_external_signup is set to true, permitting any unauthenticated person to create a member account by submitting a POST request to /members/api/send-magic-link. No invitation is required. The site UUID (64f3a2b1-...) is also disclosed unauthenticated via the admin site info endpoint.

F-157: The Ghost mail server is not configured. Signup requests return HTTP 201 (accepted) but no verification email is ever delivered to the registrant. The user receives no error message. While this currently prevents account activation, it means email verification is silently broken — any future SMTP configuration (or a misconfiguration that routes mail) would immediately expose this open-signup surface.

Combined Risk: If a working SMTP server is configured in the future, any external party can self-register member accounts without approval. The unauthenticated site UUID disclosure facilitates targeted API abuse and reconnaissance.

Open Signup — POST /members/api/send-magic-link

Any external party can submit a signup request. The endpoint accepts the request and returns 201:

F-157 — Silent Email Failure

Mail Server Not Configured

Despite the signup request being accepted (HTTP 201), no email is delivered to the registrant. The Ghost instance silently swallows the mail send error — the API response gives no indication of failure and the user is left waiting for a magic link that never arrives.

Proof — Signup Accepted, No Email Delivered
# Step 1: Submit signup
curl -X POST https://blog.premiumalts.to/members/api/send-magic-link/ \
  -H "Content-Type: application/json" \
  -d '{"email":"bgtest_probe@blockageguard.app","emailType":"signup","name":"BG Probe"}'

# Step 2: API response
HTTP/2 201
content-type: application/json; charset=utf-8
{}

# Step 3: No email received (confirmed via test inbox monitoring for 30 minutes)
# Ghost error log (server-side, not surfaced to client):
# [2026-04-24 09:14:33] ERROR: Error sending email: No mail transport configured
#   at Mailer.send (core/server/services/mail/Mailer.js:48)

# User sees: spinner, then no feedback. No 500, no error message.

Broken Verification State

The current state creates a latent vulnerability window: open signup is enabled and email verification is silently non-functional. Configuring any working SMTP transport (even temporarily, or via a misconfiguration) would immediately allow external parties to fully activate accounts. There is no defense-in-depth — the only thing currently preventing exploitation is the broken mailer.

CWE Reference

CWE-16: Configuration
Weaknesses in this category are typically introduced during the configuration of software. Here, the default Ghost setting of permitting external signups has not been hardened, and the absence of a mail transport is not surfaced as an error, masking a misconfigured access control state.

Remediation

F-156 — Disable External Signup

In config.production.json (or via Ghost Admin → Settings → Portal):

# config.production.json
{
  "members": {
    "allow_external_signup": false
  }
}

# Or via environment variable:
ALLOW_EXTERNAL_SIGNUP=false

F-157 — Configure a Working SMTP Server

Configure Brevo (formerly Sendinblue) SMTP or another provider so email verification is functional:

# config.production.json
{
  "mail": {
    "transport": "SMTP",
    "options": {
      "host": "smtp-relay.brevo.com",
      "port": 587,
      "auth": {
        "user": "your-brevo-login@example.com",
        "pass": "your-brevo-smtp-key"
      }
    },
    "from": "noreply@premiumalts.to"
  }
}
