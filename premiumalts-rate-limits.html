Overview

Four distinct API endpoints on premiumalts.to have no rate limiting whatsoever. Repeated automated requests to each endpoint never produced an HTTP 429 response, confirming the absence of any throttling, lockout, or CAPTCHA enforcement. This enables credential stuffing, coupon brute-forcing, credit code enumeration, and email bombing at full throughput.

F-014 — Login Endpoint: No Rate Limiting

20 sequential login attempts with different password combinations were sent to POST /api/auth/login. All 20 returned HTTP 401 (wrong credentials). No lockout, no CAPTCHA, no 429.

done

Remediation

All four endpoints require server-side rate limiting. The recommended approach for a Spring Boot application stack is the Bucket4j library integrated via a servlet filter or Spring interceptor. For each endpoint:

Finding Endpoint Recommended Limit Window
F-014 POST /api/auth/login 5 attempts per IP 15 minutes, then exponential backoff + CAPTCHA
F-015 GET /coupon/validate/{code} 10 requests per IP 1 minute
F-016 POST /api/me/credit/redeem 5 attempts per authenticated user 1 hour
F-024 POST /api/auth/forgot-password 3 requests per IP and 3 per email 1 hour

Spring Boot / Bucket4j Example

// build.gradle
implementation 'com.github.vladimir-bukhtoyarov:bucket4j-core:7.6.0'

// RateLimitInterceptor.java
@Component
public class RateLimitInterceptor implements HandlerInterceptor {
    private final Cache<String, Bucket> cache = Caffeine.newBuilder()
        .expireAfterWrite(1, TimeUnit.HOURS).build();

    @Override
    public boolean preHandle(HttpServletRequest req, ...) {
        String key = req.getRemoteAddr() + ":" + req.getRequestURI();
        Bucket bucket = cache.get(key, k ->
            Bucket.builder()
                .addLimit(Bandwidth.classic(10, Refill.greedy(10, Duration.ofMinutes(1))))
                .build());
        if (bucket.tryConsume(1)) return true;
        response.setStatus(429);
        return false;
    }
}
Additional controls: Implement account lockout after N failed login attempts (F-014). Use a consistent response time for forgot-password regardless of whether the account exists, to prevent account enumeration. Add Retry-After headers on 429 responses per RFC 6585.
