Overview

Two critical vulnerabilities in the Icenox checkout completion flow allow an attacker to steal any user's pending order. F-155a shows that any authenticated user can complete another user's checkout by UUID. F-155b combines F-155a with the unauthenticated PATCH from F-0a to produce a fully unauthenticated end-to-end order hijack — no credentials required at any step.

F-155a — Cross-User Checkout Completion (Authenticated)

The endpoint POST /checkout/{id}/complete does not validate that the authenticated user owns the checkout being completed. Any logged-in account can complete any other user's checkout by supplying the victim's UUID — obtained trivially via the unauthenticated GET (F-000).

F-155b — Full Unauthenticated Order Hijack (F-000 + F-0a + F-155a)

By combining three findings — unauthenticated checkout read (F-000), unauthenticated customer PATCH (F-0a), and cross-user completion (F-155a, which also works unauthenticated when combined with F-0a) — an attacker can execute a complete order theft without any account or credentials.

This chain requires zero authentication at every step. The attacker only needs to know (or enumerate) a victim's checkout UUID.

Remediation

Finding Recommendation Priority
F-155a Require authentication on POST /checkout/{id}/complete. Extract userId from the JWT and verify it matches the checkout's ownerId before proceeding. Reject with HTTP 403 if they do not match. Critical
F-155b F-155b is only possible because F-0a (unauthenticated PATCH /customer) and F-000 (unauthenticated GET) exist. Fix all three root causes: add auth guards on GET, PATCH, and POST /complete. No single fix is sufficient — all three must be closed. Critical
Both Implement a central checkout ownership validation helper used by all /checkout/{id}/* route handlers. The helper should: (1) require a valid JWT, (2) look up the checkout by UUID, (3) assert jwt.sub == checkout.userId, and (4) return 403 on mismatch. Do not rely on UUID secrecy as a security control. Critical
Defense-in-depth Transition checkout IDs to opaque tokens bound to a session or user at creation time, rather than standalone UUIDs. Log and alert on any attempt to complete a checkout where the authenticated user does not match the checkout owner. High
Related findings: See F-000 / F-0a — Checkout IDOR + Unauthenticated Customer Patch for the prerequisite vulnerabilities that enable the unauthenticated chain.
