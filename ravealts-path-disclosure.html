Why This Matters

These paths reveal:

  • The server's internal db/ directory structure
  • Naming conventions for account credential flat files
  • Which product categories use flat files vs directories (confirming storage architecture)
  • Space in filename (db/Premium unban.txt) — hints at manual file management

Remediation

Fix: Remove stock_file and stock_directory fields from the public API response entirely. These are implementation details only needed server-side. The API consumer (the frontend) has no legitimate use for these paths. 
// In the products serializer/controller, use a projection:
const publicProduct = {
  id: product.id,
  name: product.name,
  description: product.description,
  price: product.price,
  category: product.category,
  stockCount: product.stockCount,
  restockSoon: product.restockSoon,
  // REMOVE: stock_file, stock_directory
};

F-162 — Missing Authentication on POST /api/alts/purchase/get-account-data

Finding ID
F-162
Severity
HIGH
Target
api.ravealts.com
Endpoint
POST /api/alts/purchase/get-account-data
CWE
CWE-306: Missing Authentication
CVSS (est.)
7.5 HIGH
⚠️ Auth Bypass: The authentication check only fires when an API key IS present in the request. When no API key is provided at all, the check is silently skipped. The /api/alts/ strict path (which should return proper HTTP 401) also returns HTTP 200 on unauthenticated requests to this endpoint.

Behavior Comparison

Auth StateError TypeHTTP StatusExpected
No auth key (no header, no body field) invalid_input 200 Should be 401
Invalid API key in Authorization header invalid_api_key 401 Correct
Invalid API key in auth_key body field invalid_api_key 401 Correct
No auth — strict path (/api/alts/) invalid_input 200 Should be 401

Proof of Concept

# Auth check triggered (correct — 401 returned):
curl -s -X POST https://api.ravealts.com/api/alts/purchase/get-account-data \
  -H "Authorization: Bearer INVALID_KEY" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com"}'

→ HTTP 401  {"status":"error","error_type":"invalid_api_key","message":"Invalid API key"}

# Auth check bypassed (BUG — 200 returned on strict path):
curl -s -X POST https://api.ravealts.com/api/alts/purchase/get-account-data \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com"}'

→ HTTP 200  {"status":"error","error_type":"invalid_input","message":"Invalid request"}
           ↑ Auth bypassed. Fails at a later validation step — not at the auth layer.

Likely Root Cause

// Pseudocode — probable vulnerable pattern:
const authKey = req.headers['x-api-key'] || req.body?.auth_key;
if (authKey && !validateApiKey(authKey)) {
  // ✅ Only checks when key IS present
  return res.status(401).json({ error_type: 'invalid_api_key' });
}
// ⛔ Reaches here when authKey is undefined — auth skipped
processRequest(req, res);

Remediation

Fix: Require the API key to be present before ANY processing. Fail closed, not open. 
// Correct pattern — fail when absent OR invalid:
const authKey = extractApiKey(req);  // from headers + body
if (!authKey || !validateApiKey(authKey)) {
  return res.status(401).json({ error_type: 'invalid_api_key', message: 'Invalid API key' });
}
// Only reaches here with a valid key
processRequest(req, res);
Additionally: Apply a middleware-level auth guard to all API routes under /alts/ and /api/alts/ rather than inline checks per endpoint. This ensures consistent enforcement across all current and future routes.
